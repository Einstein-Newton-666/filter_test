图优化架构

本页只讲当前代码的图优化结构。入口节点负责 ROS 和 TF；业务图模型负责目标几何和因子； auto_graph::GraphOptimizer 只负责 typed 变量、GTSAM key、初值、因子和求解器生命周期。

当前架构 图优化架构 因子图原理 GTSAM 指南 ROS2 版本参考
核心边界：GraphOptimizer 不知道 ROS topic、相机话题、装甲板编号、 rune 叶片，也不做目标匹配。它只提供“把 typed 变量和 factor 写进 GTSAM”的通用能力。

2. typed core 怎么工作

declareDynamic<T>("name", prefix)  -> 每帧一个 key
declareStatic<T>("name", prefix)   -> 固定 frame 0 的 key

beginInit()
  addPrior(var, value, noise)
finishInit()

beginFrame()
  insert(var, predicted_value)
  addFactor<FactorType>(...)
solve()
  -> iSAM2.update(...) 或 fixed-lag smoother update(...)
estimate(var)
能力说明
typed variableVar<gtsam::Point3>、Var<gtsam::Rot2>、Var<double> 分类型使用，减少 key/type 混用。
动态变量frame 递增时 key 变化，用于中心、速度、yaw、roll 等时间状态。
静态变量key 固定在 frame 0，用于半径、dz、normal_yaw；fixed-lag 下刷新 timestamp，避免被边际化。
辅助 key每帧每个装甲板或叶片的 Pose3，用于拆开像素观测和几何约束。
求解器smoother_lag == 0 使用 iSAM2；大于 0 使用 fixed-lag smoother。

3. 装甲板图模型

状态变量

变量类型动态含义
centerPoint3是目标中心位置
velocityVector3是中心速度
yawRot2是目标朝向
vyawdouble是yaw 角速度
radius_a/radius_bdouble否奇偶装甲板半径的 logistic 无界状态
dzdouble否奇偶装甲板高度差

单帧写图流程

1. 用上一帧状态做常速预测，插入 center/velocity/yaw/vyaw 初值
2. 写运动因子:
   TranslationFactor, VelocityFactor, YawFactor, VyawFactor
3. 对观测装甲板做 unique match
4. 每个匹配装甲板:
   - 插入辅助 Pose3 key
   - 加 Pose3 prior 作为局部锚点
   - 对四个角点写 ArmorTypedReprojFactor
   - 写 ArmorRadiusCenterZFactor 或 ArmorRadiusDZFactor
   - 四角点齐全时写 ArmorEdgeCenterZReprojFactor 或 ArmorEdgeDZReprojFactor
5. solve 后读取 center/yaw/radius/dz，生成预测装甲板和 tracker target
因子约束
TranslationFactorcenter_k = center_{k-1} + velocity_{k-1} * dt
YawFactoryaw_k = yaw_{k-1} + vyaw_{k-1} * dt，用 Rot2 处理角度 wrap。
ArmorTypedReprojFactor辅助 Pose3 的局部角点投影后应落在观测像素上。
ArmorRadiusCenterZFactor偶数装甲板 pose 与中心、yaw、半径一致。
ArmorRadiusDZFactor奇数装甲板 pose 与中心、yaw、半径和 dz 一致。
ArmorEdgeCenterZReprojFactor偶数装甲板从 radius_a/yaw/center 直接预测四边，使用 auto_aim EdgeLoss 风格残差。
ArmorEdgeDZReprojFactor奇数装甲板从 radius_b/dz/yaw/center 直接预测四边，补充 2D 边缘约束。

direct edge 残差每条边 1 维：顶点位移和边长差按观测边长归一化，边方向用 atan2(cross,dot) 计算角度差，再用装甲板倾斜角在长度项和角度项之间加权。

4. 能量机关图模型

变量类型动态含义
centerPoint3是能量机关中心
rollRot2是当前叶片角度
vrolldouble是roll 角速度
normal_yawRot2否风车平面法线 yaw
每个有效 RuneTarget:
  resolve blade_index (new compact input; legacy 5-slot fallback by array index)
  auxiliary Pose3 key: q/s/t/u/y + frame
  5-point PnP pose prior
  5 x RuneBladeReprojFactor
  1 x RuneBladeGeometryFactor
  1 x RuneBladeDirectReprojFactor
因子约束
RuneRollFactorroll_k = roll_{k-1} + vroll_{k-1} * dt
RuneVrollFactorvroll_k - vroll_{k-1} 随机游走。
RuneBladeReprojFactor叶片五点重投影误差。
RuneBladeGeometryFactor辅助叶片 pose 与 center/roll/normal_yaw 的位置、roll、yaw 一致。
RuneBladeDirectReprojFactor直接把四个扇叶点的对角线角度/长度残差写到 center/roll/normal_yaw。
`rune_graph_optimizer` 用图优化后的连续 roll 序列拟合大符 5 参数曲线；数据不足时退回常速 roll 外推。

6. 修改规则

  • 新增业务模型时优先新增模型文件，不要把 ROS 或相机语义放进 graph_core。
  • 新因子必须先写残差为零的测试；涉及角度时覆盖 wrap 场景。
  • 静态变量进入 fixed-lag smoother 时要确认 timestamp 刷新，否则会被边际化。
  • 调参优先改 YAML，不要先改头文件默认值。
  • 匹配和观测拒绝逻辑属于业务图模型，不属于通用 core。
