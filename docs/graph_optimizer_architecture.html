template<typename Model>
class GraphModelRunner {
public:
    void initialize(const typename Model::InitInput& input) {
        InitContext ctx(spec_.layout());
        Model::initialize(ctx, input, variables_);      // ← Model 写 frame 0 初值和 prior
        auto initial = ctx.build();
        optimizer_.initialize(spec_.layout(), initial.values, initial.graph);
    }

    typename Model::Output update(const typename Model::FrameInput& input) {
        auto frame = optimizer_.beginFrame(input.dt);    // ← 创建本帧 FrameContext
        Model::addFrame(frame, input, variables_);       // ← Model 写初值和全部因子
        auto solve_result = optimizer_.solve();          // ← optimizer 执行 ISAM2
        OutputContext out(optimizer_, solve_result, spec_.layout());
        return Model::makeOutput(out, input, variables_);
    }                                                    // ← Model 提取后验
};

4. ModelSpec 与 Var — 类型安全的变量注册

旧版 VariableLayout 使用 VectorXd 分块索引，手动管理偏移量容易出错。新版模型通过 ModelSpec 声明变量，并用 Var<T> 句柄在后续代码中引用变量：

// ArmorCvPixelProblem::describe() — 声明 7 组变量并返回 typed handle
struct Variables {
    auto_graph::Var<gtsam::Point3> center;
    auto_graph::Var<gtsam::Vector3> velocity;
    auto_graph::Var<gtsam::Rot2> yaw;
    auto_graph::Var<double> vyaw;
    auto_graph::Var<double> radius_a;
    auto_graph::Var<double> radius_b;
    auto_graph::Var<double> dz;
};

static Variables describe(auto_graph::ModelSpec& spec) {
    return {
        spec.dynamic<gtsam::Point3>("center", 'x'),
        spec.dynamic<gtsam::Vector3>("velocity", 'v'),
        spec.dynamic<gtsam::Rot2>("yaw", 'r'),
        spec.dynamic<double>("vyaw", 'w'),
        spec.staticVar<double>("radius_a", 'a'),
        spec.staticVar<double>("radius_b", 'b'),
        spec.staticVar<double>("dz", 'z')};
}
变量名字GTSAM KeyC++ 类型维度动态/静态
中心位置centerSymbol('x', k)Point33D每帧
中心速度velocitySymbol('v', k)Vector33D每帧
朝向yawSymbol('r', k)Rot21D (流形)每帧
角速度vyawSymbol('w', k)double1D每帧
半径 aradius_aSymbol('a', 0)double1D (logistic)静态
半径 bradius_bSymbol('b', 0)double1D (logistic)静态
高度差dzSymbol('z', 0)double1D静态
类型安全：FrameContext::insert(vars.center, val) 从 Var<T> 得到变量名和类型。 如果底层 layout 类型不一致，会抛出 runtime_error("variable 'center' requested with wrong type")。

4.1 FrameContext 的使用约定

FrameContext 是 Model 构建单帧图的入口。它不拥有优化器，只把本帧的初值和因子追加到 GraphOptimizer 的暂存区。

接口用途典型使用位置
insert(var, value) 插入 ModelSpec 中注册过的主变量初值，并检查 C++ 类型 ArmorCvPixelProblem::addFrame 插入 center/velocity/yaw/vyaw
insertKey<T>(key, value) 插入 Problem 自己生成的辅助变量，通常不在 layout 中注册 给每块匹配装甲板插入 Pose3 key
addPrior(var, value, noise) 给 layout 主变量添加 prior 简单测试或初始化图
addPriorKey<T>(key, value, noise) 给自定义辅助 key 添加 prior 给装甲板 Pose3 添加局部锚点
addFactor<Factor>(...) 追加任意 GTSAM factor 运动因子、像素因子、几何因子 
void ArmorCvPixelProblem::addFrame(auto_graph::FrameContext& graph,
                                  const FrameInput& input,
                                  const Variables& vars) {
    auto& state = input.runtime.state;
    state.center += state.velocity * input.dt;
    state.yaw += state.vyaw * input.dt;

    graph.insert(vars.center, toPoint3(state.center));
    graph.insert(vars.velocity, state.velocity);
    graph.insert(vars.yaw, gtsam::Rot2::fromAngle(state.yaw));
    graph.insert(vars.vyaw, state.vyaw);

    const uint64_t k = graph.frameId();
    graph.addFactor<TranslationFactor>(
        translation_noise,
        graph.key("center", k - 1), graph.key("velocity", k - 1),
        graph.key("center", k), input.dt);
}

7. 几何约束因子详解

7.1 装甲板几何关系

4 块装甲板围绕中心 (俯视图), 相差 90°:

          armor1 (r2, dz)
               │
    armor2 ─── ● ─── armor0 (r1)
    (r1)        │        (r1)
             armor3 (r2, dz)

armor_position = center - r × [cos(armor_yaw), sin(armor_yaw)]
armor_yaw      = center_yaw + index × 90°

偶数装甲板 (0,2): 使用 radius_a (r1), 与中心同层 (z=0)
奇数装甲板 (1,3): 使用 radius_b (r2), 比中心高 dz

7.2 四个误差分量的物理含义

分量公式期望值物理含义控制变量
切向 tx·dx + ty·dy ≈ 0 中心在装甲板宽度方向无偏移。tx,ty 是装甲板切线方向 (法线逆时针 90°) center.x, center.y
径向 nx·dx + ny·dy - r = r 中心沿法线到装甲板的距离 = logistic 编码的半径 center.x, center.y, radius
z center_z - armor_z
center_z + dz - armor_z		 = 0 / dz 偶数装甲板与中心同高；奇数装甲板比中心高 dz center.z, dz
yaw armor_yaw ⊖ pred_yaw ≈ 0 装甲板朝向 = center_yaw + index·90°。用 Rot2::localCoordinates (SO(2) 流形减法) center_yaw

7.3 logistic 半径编码

物理半径 r ∈ [r_min, r_max] = [0.10, 0.60] m

无界优化状态 u ∈ (-∞, +∞)
         ↓ logisticFunction
物理半径 r = r_min + (r_max - r_min) / (1 + exp(-u))

雅可比链: ∂error/∂u = ∂error/∂r · ∂r/∂u
         ∂r/∂u = (r_max - r_min) · s · (1-s)   其中 s = (r-r_min)/(r_max-r_min)

当 r → r_min 或 r → r_max 时: ∂r/∂u → 0
  → 优化器自动"软约束"半径不越界，不需要硬截断

7.4 Pose3 H1 4×6 雅可比

当前 ArmorRadiusCenterZFactor 和 ArmorRadiusDZFactor 都对装甲板 Pose3 提供完整 H1，维度为 4×6。6 列对应 GTSAM Pose3::retract 的局部扰动顺序： 前三列是旋转扰动，后三列是平移扰动。 

H1 = [
  ∂tangential / ∂rot(3), ∂tangential / ∂trans(3)
  ∂radial     / ∂rot(3), ∂radial     / ∂trans(3)
  ∂z          / ∂rot(3), ∂z          / ∂trans(3)
  ∂yaw        / ∂rot(3), ∂yaw        / ∂trans(3)
]

实现位于 armor_model.cpp::armorPoseGeometryJacobian()。平移部分通过 J_position * armor_pose_odom.rotation() 把 Pose3 局部平移扰动映射到 odom 系； 旋转部分通过 RPY yaw 对局部角速度的链式导数影响 tangential/radial/yaw 残差。

验证方式：单元测试 RadiusCenterZFactorPoseJacobianMatchesFiniteDifference 和 RadiusDZFactorPoseJacobianMatchesFiniteDifference 使用非单位 T_camera_to_odom 对 H1 的 6 列逐列做有限差分检查。

8. 求解器架构 (3 种后端)

后端触发条件适用场景性能
ISAM2 smoother_lag ≤ 0 无限历史, 离线精度最高 增量 O(m'), m'=受影响变量数
IncrementalFixedLagSmoother smoother_lag > 0 && type=incremental 固定窗口, 在线实时 增量 + 边缘化 O(window)
BatchFixedLagSmoother smoother_lag > 0 && type=batch 固定窗口, LM 批量优化 批量 LM, 更慢但更鲁棒
静态 key 重打时间戳： FixedLagSmoother 按时间戳边缘化旧变量。 radius_a / radius_b / dz 是静态参数（只有 frame 0 一个 key）， 如果不每帧刷新时间戳，它们会在 smoother_lag 帧后被边缘化 → 几何约束失去连接 → 优化发散。solve() 中显式重打时间戳解决此问题。

8.1 cold start 与失败处理

阶段GraphOptimizer 行为Tracker 行为
第一帧初始化 Problem::initialize 创建 frame 0 初值和 prior 图，不执行当前帧运动边 accepted_frame=true，solved=false，输出基于初始化状态的预测装甲板
cold start cold_start_count_ < cold_start_frames 时累计初值和因子，不提前优化 保持预测状态连续输出，cold_start=true
正常优化 iSAM2 或 fixed-lag smoother 更新 latest_estimate_ makeOutput 用 estimate 覆盖 runtime 状态，并提交 last armor index
求解失败 SolveResult.failed=true，错误文本写入 error_message 不 reset 生命周期，只把失败状态返回给发布层
所有观测都未匹配 本帧没有有效装甲板几何连接 ArmorGraphTracker::reset()，下一帧重新初始化，避免错匹配污染半径和 dz

s2qxy / s2qz / s2qyaw TrackerConfig → addMotion 位置/yaw 积分运动因子的过程噪声方差，代码中会转为 sigma
vel_sigma / vyaw_sigma VelocityFactor / VyawFactor 速度和角速度随机游走强度，越小越平滑，越大越容易跟随突变
match_max_cost matchArmorIndicesUnique 匹配门控阈值；过大的观测会被标记为 -1，全未匹配帧触发 reset
pixel_noise_std ArmorTypedReprojFactor 角点像素重投影残差噪声，单位 px
geo_tangential / geo_radial / geo_height / geo_yaw ArmorRadiusCenterZFactor / ArmorRadiusDZFactor 几何约束四维残差噪声；半径和 dz 收敛很依赖这些权重
camera_fx/fy/cx/cy 与畸变参数 ArmorTypedReprojFactor GTSAM Cal3DS2 投影模型使用的相机内参
cold_start_frames GraphOptimizerConfig 前 N 帧只累计因子，不优化；用于给 iSAM2 更稳定的初始图
smoother_lag / smoother_type GraphOptimizer 选择纯 iSAM2 或 fixed-lag smoother；smoother_lag <= 0 时使用 iSAM2

10.3 在 C++ 中直接使用 ArmorGraphTracker

ArmorGraphTracker 是推荐给 ROS node 或上层业务调用的接口。它隐藏 ArmorCvPixelGraph、runner/runtime、初始化和 reset 细节。

#include "filter_test/graph_optimizer/armor_tracker.hpp"

filter_test::graph_optimizer::TrackerConfig config;
config.optimizer.cold_start_frames = 3;
config.match_max_cost = 1.5;

filter_test::graph_optimizer::ArmorGraphTracker tracker(config);

filter_test::graph_optimizer::TrackerFrameInput input{
    armors_msg,           // const auto_aim_interfaces::msg::Armors&
    dt,                   // double, seconds
    T_camera_to_odom      // Eigen::Isometry3d
};

auto result = tracker.update(input);
if (!result.accepted_frame) {
    // 空帧或全未匹配帧。全未匹配时 tracker 内部已 reset，下帧会重新初始化。
    return;
}
if (result.solve_failed) {
    // 可读取 result.solve_error；当前 tracker 生命周期仍保留。
    return;
}

const auto& state = result.state;
const auto& armors = result.predicted_armors;

10.4 新增自定义模型

新增模型时不要改 GraphOptimizer。创建新的 Model 类型，实现与 ArmorCvPixelProblem 相同的 describe/initialize/addFrame/makeOutput 接口，然后在 graph facade、tracker 或测试中接入 GraphModelRunner。

struct MyModel {
    using InitInput = MyInitInput;
    using FrameInput = MyFrameInput;
    using Output = MyOutput;

    struct Variables {
        auto_graph::Var<gtsam::Point3> center;
    };

    static Variables describe(auto_graph::ModelSpec& spec);
    static void initialize(auto_graph::InitContext& ctx,
                           const InitInput& input,
                           const Variables& vars);
    static void addFrame(auto_graph::FrameContext& graph,
                         const FrameInput& input,
                         const Variables& vars);
    static Output makeOutput(const auto_graph::OutputContext& ctx,
                             const FrameInput& input,
                             const Variables& vars);
};

auto_graph::GraphModelRunner<MyModel> runner(config.optimizer);
拆分建议：如果新模型仍是装甲板跟踪但观测不同，优先新建 armor_xxx_model.hpp/cpp； 如果是完全不同对象，保留 graph_core，新建独立的 xxx_model 和薄 tracker。

12. 与 jlu_tracker 的代码路径对比

两个 tracker 共享相同的因子图结构和 GTSAM 后端，但代码组织不同：

jlu_tracker新版 graph_optimizer
入口 jlu_tracker_node.cpp graph_optimizer_test.cpp
跟踪器类 jlu::RobotTarget — 单体类，包含所有逻辑 ArmorGraphTracker — 薄壳层, 委托给 ArmorCvPixelGraph
因子定义 jlu_tracker/factors.cpp graph_optimizer/armor_model.cpp
匹配算法 马氏距离 + unique match + NIS 启发式代价 + unique match
几何因子 H1 完整 4×6 (旋转+平移) 完整 4×6 (旋转+平移)
共享的数学工具： auto_graph::logisticFunction、auto_graph::logisticDerivative、 auto_graph::shortestAngularDistance、auto_graph::diagonalNoise 等定义在 graph_math.hpp 的 auto_graph 命名空间中，jlu_tracker 也通过 include 复用这些函数。
