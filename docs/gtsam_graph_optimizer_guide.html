1. 图优化到底在优化什么

图优化是把一堆观测、运动模型、几何约束都写成残差，然后一起求一个最可能的状态。 在概率语言里，它是最大后验估计；在数值计算里，它就是加权非线性最小二乘。 

x* = argmax p(x | z)
   = argmin Σ ||e_i(x)||²_Σ_i
      x     i

e_i(x)       : 第 i 个因子的残差，通常是 predicted - measured
Σ_i          : 该残差的协方差
||e||²_Σ     : e^T Σ^-1 e，也就是按不确定性加权后的误差

变量

要估计的未知量，例如机器人中心 center、速度 velocity、中心 yaw、半径和高度差。

因子

一条约束，例如“这一帧位置应等于上一帧位置加速度积分”、“某个角点重投影应落在检测像素上”。

噪声模型

告诉优化器这条约束可信到什么程度。标准差越小，残差被惩罚越重。

工程直觉：EKF/UKF 每帧把状态压成一个均值和协方差；因子图保留最近多帧变量和约束结构。 这样当新观测进来时，优化器可以重新调整多帧状态、静态几何量和辅助 pose，而不是只修正当前状态。

2. 因子图的形状

因子图是二分图：圆形变量节点只和方形因子节点相连。这个结构直接决定了雅可比矩阵的稀疏性。 GTSAM 的效率很大一部分来自它不把所有变量硬塞进一个大 dense 向量，而是按 key 和 factor 管理稀疏连接。

center(k-1) -- TranslationFactor -- center(k) -- Geometry -- armor_pose(k,i)
velocity(k-1) -- VelocityFactor -- velocity(k) radius_a radius_b dz
yaw(k-1) -- YawFactor -- yaw(k) -- Reprojection x4 -- pixels
节点类型在 GTSAM 中的对应物本项目例子
变量节点 gtsam::Values 中的一条 Key -> value x3 对应第 3 帧中心 gtsam::Point3，a0 对应静态半径状态 double
因子节点 gtsam::NonlinearFactor / NoiseModelFactorN TranslationFactor、ArmorTypedReprojFactor、ArmorRadiusDZFactor
边 因子构造函数里传入的 key 列表 YawFactor(noise, yaw(k-1), vyaw(k-1), yaw(k), dt)

3. GTSAM 的核心对象

理解 GTSAM 代码时，先把对象分成三类：变量容器、约束容器、求解器状态。

对象职责使用要点
gtsam::Key 变量的 64-bit id 只负责唯一标识，不携带类型。通常用 gtsam::Symbol('x', k) 生成便于调试的 key。
gtsam::Values 非线性变量表 插入和读取时必须类型一致，例如 insert(key, gtsam::Point3) 后用 at<gtsam::Point3> 取。
gtsam::NonlinearFactorGraph 非线性因子列表 可以 addPrior、emplace_shared<Factor>。它本身不求解，只保存约束。
gtsam::SharedNoiseModel 残差权重 常用 Isotropic::Sigma、Diagonal::Sigmas。残差维度必须和噪声维度一致。
gtsam::NoiseModelFactorN<...> N 元自定义非线性因子基类 模板参数就是连接的变量 C++ 类型；evaluateError 返回 residual 并按需写雅可比。
gtsam::ISAM2 增量式非线性优化器 每次 update(graph, values) 加入新因子和新初值，再 calculateEstimate() 取结果。
gtsam::FixedLagSmoother 固定时间窗平滑 需要每个 key 的时间戳；窗口外变量会被边缘化。静态 key 必须特殊处理，避免过早被清掉。
本项目的关键封装：TypedKeyLayout 给 gtsam::Key 补上了“变量名”和“C++ 类型”这两层工程约束， 防止手写 Symbol 和 Values::at<T> 时发生 prefix 冲突或类型取错。

4. 从非线性图到线性系统

GTSAM 每次求解时会在当前估计值附近线性化因子，得到稀疏线性最小二乘。

非线性因子:
  e_i(x)

在当前估计 x0 处线性化:
  e_i(x0 + dx) ~= e_i(x0) + J_i dx

堆叠所有因子:
  minimize ||A dx - b||^2

求出 dx 后:
  x_new = retract(x0, dx)
NonlinearFactorGraph
->
linearize(values)
->
Jacobian/Hessian
->
eliminate
->
Bayes tree
->
delta + estimate

4.1 为什么 iSAM2 快

批量优化每帧重新线性化、重新消元整张图。ISAM2 会维护 Bayes tree， 当新因子进入时只重算受影响的 clique，并按阈值决定哪些变量需要重新线性化。

Batch 优化

每帧:
  graph += new_factors
  values += new_initials
  重新线性化整图
  重新消元整图
  得到 estimate

ISAM2 增量优化

每帧:
  update(new_factors, new_initials)
  找到受影响的 Bayes tree 分支
  局部重新线性化和消元
  calculateEstimate()
本项目选择：smoother_lag <= 0 时使用普通 gtsam::ISAM2。 smoother_lag > 0 时使用 IncrementalFixedLagSmoother 或 BatchFixedLagSmoother，由 smoother_type 参数选择。

5. 最小 GTSAM 使用流程

下面是一个最小化的 GTSAM 思路示例：声明 key、填初值、加先验和运动约束、调用优化器。

#include <gtsam/geometry/Point3.h>
#include <gtsam/inference/Symbol.h>
#include <gtsam/nonlinear/ISAM2.h>
#include <gtsam/nonlinear/NonlinearFactorGraph.h>
#include <gtsam/nonlinear/PriorFactor.h>
#include <gtsam/nonlinear/Values.h>
#include <gtsam/slam/BetweenFactor.h>

gtsam::NonlinearFactorGraph graph;
gtsam::Values initial;

auto x0 = gtsam::Symbol('x', 0);
auto x1 = gtsam::Symbol('x', 1);

auto prior_noise = gtsam::noiseModel::Diagonal::Sigmas(
    gtsam::Vector3(0.1, 0.1, 0.1));
auto motion_noise = gtsam::noiseModel::Diagonal::Sigmas(
    gtsam::Vector3(0.05, 0.05, 0.05));

graph.addPrior(x0, gtsam::Point3(0.0, 0.0, 0.0), prior_noise);
graph.emplace_shared<gtsam::BetweenFactor<gtsam::Point3>>(
    x0, x1, gtsam::Point3(1.0, 0.0, 0.0), motion_noise);

initial.insert(x0, gtsam::Point3(0.0, 0.0, 0.0));
initial.insert(x1, gtsam::Point3(0.9, 0.1, 0.0));

gtsam::ISAM2 isam;
isam.update(graph, initial);
gtsam::Values result = isam.calculateEstimate();
auto x1_est = result.at<gtsam::Point3>(x1);
注意：真实跟踪里不要把 Values 当普通 map 随便重复插入同一个 key。 同一轮 update 中同一个新 key 只能有一个初值；已经在优化器内部存在的旧 key 不应再次作为新初值提交。

6. 自定义 factor 怎么写

GTSAM 自定义因子的核心是继承 gtsam::NoiseModelFactorN<T1, T2, ...>， 实现 evaluateError。模板参数决定这个因子连接哪些变量类型。 

class MyFactor
    : public gtsam::NoiseModelFactorN<gtsam::Point3, double> {
public:
    using Base = gtsam::NoiseModelFactorN<gtsam::Point3, double>;

    MyFactor(const gtsam::SharedNoiseModel& noise,
             gtsam::Key point_key, gtsam::Key scalar_key)
        : Base(noise, point_key, scalar_key) {}

    gtsam::Vector evaluateError(
        const gtsam::Point3& p,
        const double& s,
        gtsam::OptionalMatrixType H1,
        gtsam::OptionalMatrixType H2) const override {
        if (H1) *H1 = (gtsam::Matrix(1, 3) << 1.0, 0.0, 0.0).finished();
        if (H2) *H2 = gtsam::Matrix::Constant(1, 1, -1.0);
        return gtsam::Vector1(p.x() - s);
    }
};
检查项要求例子
残差维度 返回向量维度必须等于噪声模型维度 ArmorTypedReprojFactor 返回 2D 像素残差，因此用 Vector2(pixel_sigma, pixel_sigma)
雅可比尺寸 H_i 尺寸是 residual_dim x tangent_dim(variable_i) Pose3 的切空间是 6D，所以像素因子的 H 是 2x6
流形变量 不要直接把角度当普通差值处理跨 pi YawFactor 使用 Rot2::localCoordinates 计算 yaw 残差
可选雅可比 只有 if (H) 成立时才写矩阵 GTSAM 只在需要线性化时请求雅可比，误差评估时可能只要 residual

6.1 本项目三个观测 factor

Factor连接变量残差主要作用
ArmorTypedReprojFactor Pose3 armor_pose_camera 2D: projected_pixel - observed_pixel 每个角点一个因子，用 Cal3DS2 投影和畸变模型保留像素级信息。
ArmorRadiusCenterZFactor Pose3, double radius_a_u, Rot2 yaw, Point3 center 4D: tangential, radial, z, yaw 偶数装甲板几何约束，连接独立 armor pose 和主状态。
ArmorRadiusDZFactor Pose3, double radius_b_u, double dz, Rot2 yaw, Point3 center 4D: tangential, radial, z, yaw 奇数装甲板几何约束，比偶数装甲板多连接一个静态 dz。
ArmorEdgeCenterZReprojFactor double radius_a_u, Rot2 yaw, Point3 center 4D: 四条边的 auto_aim EdgeLoss 残差 偶数装甲板直接从主状态预测四角点，补强 2D 边缘几何约束。
ArmorEdgeDZReprojFactor double radius_b_u, double dz, Rot2 yaw, Point3 center 4D: 四条边的 auto_aim EdgeLoss 残差 奇数装甲板 direct edge 约束，比偶数多连接高度差 dz。

direct edge 因子不走辅助 Pose3。它把四条边的顶点位移、边长差和边方向角度差压成 4 维残差，噪声入口是 edge_reproj_sigma；过强时先增大该值或关闭 use_edge_reproj_factor 对比。

7.2 每帧执行顺序

ArmorCvPixelGraph::update(msg, dt, T_camera_to_odom):
  if not initialized:
    initializeFromFirstObservation()

  optimizer.beginFrame()
  addMotionFactors(dt)
  addObservationFactors(msg, T_camera_to_odom)
  solve_result = optimizer.solve()
  return makeOutput(solve_result)
阶段当前实现写入 GTSAM 的内容
initialize 第一帧用第一块装甲板反推中心初值，并给 frame 0 主状态加 prior Values 插入 c0/v0/r0/w0/a0/b0/z0；addPrior 加初始先验
addMotion 根据 runtime state 显式预测本帧初值 插入 center/velocity/yaw/vyaw 当前帧初值，添加 4 类运动因子
addObservation 匹配观测装甲板，生成每块装甲板的辅助 Pose3 key 添加 pose prior、4 个角点重投影因子、1 个几何因子
solve 冷启动阶段只累计；冷启动后提交到 iSAM2 或 fixed-lag smoother update(graph_, initial_values_)，然后清空本轮增量
makeOutput 优化成功后从 optimizer.estimate(var) 取最新值 更新 runtime state，输出 predicted armors 和匹配结果

7.3 项目风格伪代码

void ArmorCvPixelGraph::addMotionFactors(double dt) {
    optimizer_.insert(vars_.center, predicted_center);
    optimizer_.insert(vars_.velocity, predicted_velocity);
    optimizer_.insert(vars_.yaw, gtsam::Rot2::fromAngle(predicted_yaw));
    optimizer_.insert(vars_.vyaw, predicted_vyaw);

    optimizer_.addFactor<TranslationFactor>(
        translation_noise,
        optimizer_.keyPrev(vars_.center), optimizer_.keyPrev(vars_.velocity),
        optimizer_.key(vars_.center), dt);
}

void ArmorCvPixelGraph::addObservationFactors(...) {
    auto akey = armorPoseKey(optimizer_.getFrameId(), armor_index);
    optimizer_.insertAux(akey, armor_pose_camera);

    optimizer_.addFactor<ArmorTypedReprojFactor>(
        pixel_noise, akey, corner_local, K, distortion, observed_pixel);

    optimizer_.addFactor<ArmorRadiusCenterZFactor>(
        geo_noise, akey, optimizer_.key(vars_.radius_a),
        optimizer_.key(vars_.yaw), optimizer_.key(vars_.center),
        T_camera_to_odom, armor_index);
}

8. 求解器后端和 fixed-lag 细节

GraphOptimizer::ensureSmoother() 懒创建后端，三种路径互斥：

配置后端适用场景注意点
smoother_lag <= 0 gtsam::ISAM2 保留完整历史图，由 iSAM2 维护增量 Bayes tree 长时间运行时历史图会持续增长，但静态变量处理最简单。
smoother_lag > 0
smoother_type=incremental		 gtsam::IncrementalFixedLagSmoother 实时窗口平滑 必须给每个新 key 提供 timestamp；本项目还会刷新静态 key timestamp。
smoother_lag > 0
smoother_type=batch		 gtsam::BatchFixedLagSmoother 窗口内批量 LM 优化，便于对照调试 每帧窗口内重算，实时性通常弱于 incremental。
fixed-lag 静态变量坑：radius_a、radius_b、dz 是 frame 0 静态 key。如果 fixed-lag smoother 只给动态 key 更新时间戳，这些静态 key 会被认为太旧而被边缘化。本项目在 solve() 里显式刷新 typed_layout_.staticKeys() 的 timestamp 到当前帧。

9. 调参与排错清单

现象常见原因排查方向
IndeterminantLinearSystemException 或奇异矩阵 变量不可观、缺少 prior、某些 key 没有足够约束、factor 连接错 key 检查每个新变量是否有初值和约束；辅助 Pose3 是否至少有 prior、像素因子和几何因子。
估计抖动很大 观测噪声太小、运动噪声太大、匹配 index 抖动 增大 pixel_noise_std 或几何噪声；检查 match_max_cost 和上一帧 index 滞回。
响应太慢 运动噪声太小或 prior 太强，导致图不愿意跟随新观测 适当增大 s2qxy/s2qz/s2qyaw 或速度随机游走噪声。
yaw 在 pi 附近跳变 直接做标量角度差，没有用流形 residual 保持 yaw 为 gtsam::Rot2，factor 里用 localCoordinates。
半径跑到不合理范围 直接优化物理半径或 prior/几何噪声比例不合适 本项目使用 logistic 无界状态并映射到 [0.10, 0.60]；调 prior_noise.radius 和几何 radial 噪声。
fixed-lag 模式一段时间后丢变量 key timestamp 不完整，静态 key 被窗口边缘化 确认 initial_values_ 里的新 key 和 staticKeys() 都写入 KeyTimestampMap。
离群角点拖偏 pose 像素 residual 没有鲁棒核，错误角点权重过高 真实检测器接入后可给 ArmorTypedReprojFactor 的 noise 包一层 Huber/robust model。

10. 读代码时的路线

建议按调用方向读，不要从某个 factor 的雅可比开始硬啃。

  1. graph_optimizer_test.cpp：看 ROS 参数如何进入 TrackerConfig，以及 TF 和 topic 边界。
  2. armor_tracker.hpp/cpp：看空帧、初始化、reset、ArmorCvPixelGraph 调用和输出封装。
  3. graph_core.hpp/cpp：看 Var<T>、key 映射、初值/因子暂存和 GraphOptimizer::solve()。
  4. armor_model.hpp/cpp：看 ArmorCvPixelGraph 如何初始化、预测、匹配、写因子和输出。
  5. 最后看三个自定义观测 factor 的 evaluateError 和雅可比。
一句话总结：GTSAM 提供 key/value/factor/optimizer 的通用机制； 本项目用 Var<T> 和 auto_graph::GraphOptimizer 把这些机制包装成可复用 core； ArmorCvPixelGraph 和 RuneCvGraph 则把装甲板/能量机关几何、像素重投影和运动模型落成具体因子图。
