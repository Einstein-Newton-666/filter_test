2. 更新背后的 ROS 2 核心原理

ROS 2 的版本更新通常不是“某个命令多了一个参数”这么简单。很多变化都来自下面这条链路：

应用节点
rclcpp / rclpy
→
ROS Client Library
参数 / executor / callback
→
RCL / RMW
统一抽象层
→
DDS / Zenoh
发现、QoS、序列化、传输
→
工具链
launch / rosbag2 / RViz / CLI
Executor
executor 决定 callback 何时被取出、在哪个线程执行、是否按事件驱动唤醒。 新版本持续减少无意义的 wait set 重建和轮询，让高频感知节点更少占 CPU。
RMW
RMW 是 ROS API 和具体中间件之间的适配层。同一个 create_publisher 可以跑在 Fast DDS、Cyclone DDS、Connext 或 Zenoh 上，差异主要落在发现机制、QoS 能力、零拷贝和安全特性。
rosbag2
rosbag2 不是简单的“写文件工具”，它同时依赖类型支持、QoS、时间戳、服务/动作 introspection 和存储插件。 所以 Jazzy 之后能录 service，Kilted 之后能录 action，Lyrical 继续增强远程控制和丢包观测。
launch
launch 是运行时组合系统。参数文件、命名空间、组件容器、日志等级、事件处理和 substitution 都在这里汇合，适合管理一整套感知、定位、规划或自瞄 pipeline。

3. Lyrical Luth：2026 的最新稳定能力

Lyrical 的更新重点是“更低开销的执行、更强的可观测性、更灵活的运行时工具”。它对实时感知系统尤其有意义： 高频相机、PnP、滤波器、图优化、云台控制这些节点通常对 CPU 抖动和数据记录可靠性敏感。

Callback Group Events Executor

这是 rclcpp 的新 executor 方向。它把“不断扫描所有实体是否 ready”的模式，推进到事件队列驱动；官方说明相对传统 single/multithreaded executor 可降低 CPU 使用。

rclpy AsyncNode

Python 节点可以把 asyncio event loop 纳入 ROS 节点模型，在 service、timer、subscription callback 中自然使用 await。

rosidl::Buffer

面向高带宽数据的零拷贝/少拷贝路线，尤其适合图像、点云、GPU buffer 等 uint8[] 数据。

rosbag2 远程控制和丢包观测

录包可以通过 ROS service 控制；同时发布消息丢失统计，方便发现“以为录到了，实际丢了”的问题。

3.1 Events executor 的原理

传统 executor 会构造 wait set，把 subscription、timer、service、action 等实体放进去等待。 当实体很多、callback group 多、节点组合复杂时，重复收集实体和唤醒扫描会有额外开销。 Events executor 的思路是把 ready 事件排队，executor 从队列消费事件，再调度对应 callback。 

// Lyrical 思路示例：事件驱动 executor，具体类名按对应发行版头文件为准。
#include <rclcpp/rclcpp.hpp>

int main(int argc, char ** argv)
{
  rclcpp::init(argc, argv);
  auto node = std::make_shared<rclcpp::Node>("events_executor_demo");

  auto group = node->create_callback_group(rclcpp::CallbackGroupType::MutuallyExclusive);
  rclcpp::SubscriptionOptions options;
  options.callback_group = group;

  auto sub = node->create_subscription<std_msgs::msg::String>(
    "/debug_text", rclcpp::QoS(10),
    [](std_msgs::msg::String::ConstSharedPtr msg) {
      RCLCPP_INFO(rclcpp::get_logger("events_executor_demo"), "%s", msg->data.c_str());
    },
    options);

  // 新版可使用 EventsCBGExecutor 一类的事件 executor。
  // rclcpp::executors::EventsCBGExecutor executor;
  rclcpp::executors::MultiThreadedExecutor executor;
  executor.add_node(node);
  executor.spin();
  rclcpp::shutdown();
}

3.2 rosbag2 从“命令行工具”变成“可控记录服务”

对机器人实验来说，bag 的核心问题不是“能不能保存”，而是“关键事故前后的数据是否完整、是否知道丢没丢”。 Lyrical 的 rosbag2 服务控制、循环录制、split 命名和 message-loss observability 都围绕这个问题。 

# Lyrical: 录包过程可通过服务远程控制，适合调试站或上位机控制。
ros2 bag record /camera/image_raw /tf /detector/armors
ros2 service list | grep rosbag2

# 统计消息丢失事件，便于在高带宽图像链路中提前发现记录风险。
ros2 topic echo /events/rosbag2_messages_lost

4. Kilted Kaiju：动作 introspection、Zenoh Tier 1 与 Windows 工具链变化

Kilted 的技术重心是把 Jazzy 引入的 introspection、rosbag2 和新平台工作继续往前推。 对实际工程比较关键的点有三个：action 数据可以被工具链理解，rmw_zenoh_cpp 成为 Tier 1， Windows 安装路线转向 Pixi/Conda。

模块更新点工程意义
ros2cli ros2 action echo 等 action introspection 能力 调试导航、机械臂、自瞄行为树时，可以像 echo topic 一样观察 action goal/result/feedback。
rosbag2 支持记录和回放 action 数据，支持按 sent/received 顺序播放等选项 能完整复现“发送目标、反馈、取消、结果”的行为序列，不再只靠 topic 侧面推断。
rclpy ActionClient/ActionServer 类型标注、pub/sub/client/server 泛型支持 Python 项目可以用 mypy/ament_mypy 更早发现类型错误。
rmw_zenoh_cpp 进入 Tier 1 Zenoh 从实验路线变成官方重点支持的 RMW 之一，适合跨网段、边缘网络和非纯 DDS 网络探索。
rosidl_rust Rust IDL generator 进入默认代码生成器列表 为 Rust 节点和工具链生态铺路。

4.1 为什么 action 录包比 topic 录包复杂

Action 在 ROS 2 里不是单个 topic，而是一组 service + topic 的协议组合： goal service、result service、cancel service、feedback topic、status topic。 rosbag2 要录 action，必须理解这些实体属于同一个 action 名称，并能在回放时重建时序关系。 

# Kilted: action introspection 方向
ros2 action list
ros2 action info /navigate_to_pose
ros2 action echo /navigate_to_pose

# Kilted: rosbag2 支持 action 数据记录/回放
ros2 bag record --all
ros2 bag info ./my_bag --sort name
ros2 bag play ./my_bag --message-order sent

5. Jazzy Jalisco：现代 LTS 基线与 Gazebo 集成重整

Jazzy 是 Humble 之后最重要的 LTS 基线。它的更新不只是包版本升级，还明确改变了 ROS 2 与 Gazebo 的集成方式： 每个 ROS 2 发行版对应推荐 Gazebo 发行版，Jazzy 推荐 Gazebo Harmonic，并提供 gz_common_vendor、gz_math_vendor、gz_sim_vendor、sdformat_vendor 等 vendor 包，方便 ROS 包通过 rosdep/apt/CMake 消费 Gazebo 依赖。

对仿真项目的影响：如果你的仿真节点依赖 Gazebo 库，Jazzy 之后更推荐依赖 ROS 发行版内的 gz_*_vendor 包，而不是让每个项目自己追 Gazebo upstream 的版本组合。

5.1 Jazzy 重点更新

  • image_transport 支持自定义 QoS，并可在运行时禁用特定 plugin。
  • message_filters 支持 Type Adaptation，过滤链可以处理适配后的类型。
  • rcl 增加 type description service，节点可暴露自身类型描述，契合 REP 2016。
  • rclpy 增加 ParameterEventHandler，Python 节点更容易监听参数变化。
  • rosbag2 支持 service recording/playback，MCAP 中可保留更完整时间戳和 metadata。
  • rviz2 增强 TF frame 过滤、时间重置、显示能力。
  • rclcpp executor 修复数据竞争，并调整 wait set 使用方式。

5.2 Service introspection 和 rosbag2 的关系

Topic 是持续数据流，天然适合录包；service 是请求-响应 RPC，传统 bag 只录 topic 无法完整复盘一次服务调用。 Jazzy 的 service recording/playback 建立在 Service Introspection 之上，能够把 request/response 也纳入记录。 

# Jazzy: 记录 service 数据
ros2 bag record --all-services

# 同时记录 topic 和 service
ros2 bag record --all --all-services

# 回放包含 service 数据的 bag
ros2 bag play ./service_debug_bag

5.3 Type description 为什么重要

早期工具要理解消息类型，通常依赖本机已经安装对应 interface package。 Type description service 让节点能够提供“我发布/订阅/服务使用的类型长什么样”的描述。 这有利于动态工具、跨语言绑定、bag metadata 和运行时 introspection。 

// Jazzy 思路示例：工具可以通过节点提供的 get_type_description 服务理解接口类型。
// 实际服务由 rcl/rclcpp/rclpy 节点基础设施提供，应用侧通常不需要手写服务端。
// 对调试工具、录包工具、桥接工具来说，这比只知道 "std_msgs/msg/String" 更有信息量。

6. Humble Hawksbill：你当前工作区的基础能力

Humble 是 Ubuntu 22.04 上最常见的 ROS 2 LTS，也是本工作区的发行版。 它已经具备很多现代 ROS 2 能力：Content Filtered Topics、launch 参数增强、SROS2 CRL、 rosbag2 snapshot/burst/convert、loaned message 控制、Python lifecycle node 等。

6.1 Humble 对 C++ 包结构的影响

Humble release notes 明确提到 C++ headers 安装到子目录。这也解释了本工作区的约定： 头文件布局使用 include/<package>/...，引用写成 "filter_test/..."。 这能避免多个包把同名头文件安装到同一 include 根目录下产生冲突。 

# Humble 及之后推荐的包内头文件布局
target_include_directories(graph_optimizer_test
  PUBLIC
    $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
    $<INSTALL_INTERFACE:include>)

install(DIRECTORY include/
  DESTINATION include)

6.2 Content Filtered Topics 的原理

普通 subscription 会把 topic 上的所有样本交给订阅者，应用层再丢弃不关心的数据。 Content Filtered Topics 把过滤表达式下推到中间件：如果 RMW 支持，订阅端可以声明“只要满足条件的样本”。 这能降低网络传输、反序列化和 callback 压力。 

// Humble 已引入 CFT 概念；具体 API 和 RMW 支持情况需要按发行版确认。
// 思路：订阅方声明过滤表达式，让底层 DDS/RMW 尽量只传递匹配样本。
auto qos = rclcpp::QoS(10).reliable();
auto sub = node->create_subscription<auto_aim_interfaces::msg::Armors>(
  "/detector/armors", qos,
  [](auto_aim_interfaces::msg::Armors::ConstSharedPtr msg) {
    // 如果底层不支持 CFT，仍然可以在 callback 中做应用层过滤。
    for (const auto & armor : msg->armors) {
      if (armor.number == "3") {
        // 只处理目标装甲板
      }
    }
  });

7. Rolling Ridley：提前适配下一代 ROS 2

Rolling 是持续滚动开发版。新功能通常先进入 Rolling，经过一段时间的 ABI/API 冻结、测试和平台同步后， 再成为下一个稳定发行版的一部分。它适合以下场景：

  • 你维护的包要提前兼容下一版 ROS 2。
  • 你需要验证新 RMW、新 executor、新 rosbag2 功能是否解决当前瓶颈。
  • 你要向 ROS 2 上游提交修复，必须在最新开发分支复现问题。
工程建议：生产机器人不要直接依赖 Rolling。可以用 Rolling 建独立 CI job 或容器， 做“提前发现未来迁移问题”的验证，而不是替代 Humble/Jazzy/Lyrical 的部署环境。

8. 具体代码和配置示例

8.1 QoS：传感器流和可靠控制流要分开

ROS 2 的 QoS 是 DDS/RMW 能力向应用层暴露的核心接口。图像、点云这类高频数据通常倾向 SensorDataQoS；控制命令、目标状态、参数事件通常更关注可靠性。 

auto image_sub = node->create_subscription<sensor_msgs::msg::Image>(
  "/camera/image_raw",
  rclcpp::SensorDataQoS(),
  [](sensor_msgs::msg::Image::ConstSharedPtr image) {
    // 高频图像：允许丢旧帧，减少排队延迟。
  });

auto target_pub = node->create_publisher<auto_aim_interfaces::msg::TrackerTarget>(
  "/tracker/target",
  rclcpp::QoS(10).reliable().transient_local());

8.2 Callback group：把耗时计算和控制回调隔离

在装甲板跟踪中，PnP、EKF/UKF、GTSAM 更新可能耗时；云台控制 timer 又希望稳定触发。 可以用 callback group 把不同工作分组，再用 MultiThreadedExecutor 或新版本 executor 调度。 

auto heavy_group =
  node->create_callback_group(rclcpp::CallbackGroupType::MutuallyExclusive);
auto control_group =
  node->create_callback_group(rclcpp::CallbackGroupType::MutuallyExclusive);

rclcpp::SubscriptionOptions sub_options;
sub_options.callback_group = heavy_group;

auto armors_sub = node->create_subscription<auto_aim_interfaces::msg::Armors>(
  "/detector/armors", rclcpp::SensorDataQoS(),
  [this](auto_aim_interfaces::msg::Armors::ConstSharedPtr msg) {
    tracker_.update(*msg);  // 可能包含滤波或图优化
  },
  sub_options);

auto timer = node->create_wall_timer(
  std::chrono::milliseconds(5),
  [this]() { publish_control_command(); },
  control_group);

8.3 参数 descriptor：把配置约束写进节点

Lyrical 增强了数组参数的范围检查；但在 Humble/Jazzy 中，标量参数 descriptor 已经很实用。 它让 ros2 param describe 能看到约束，也能在 set 参数时做边界校验。 

rcl_interfaces::msg::ParameterDescriptor desc;
desc.description = "Pixel observation noise standard deviation.";

rcl_interfaces::msg::FloatingPointRange range;
range.from_value = 0.1;
range.to_value = 20.0;
range.step = 0.0;
desc.floating_point_range.push_back(range);

double pixel_sigma = node->declare_parameter("pixel_sigma", 2.0, desc);

8.4 Launch：参数文件、日志等级和组合节点

launch 的价值是把“怎么运行系统”变成可审查的代码。Humble 已支持大量 launch_ros 能力， Lyrical 又增加了 XML/YAML launch substitutions 和 per-message log severity。 

# launch/vision_debug.launch.py
from launch import LaunchDescription
from launch.actions import DeclareLaunchArgument
from launch.substitutions import LaunchConfiguration
from launch_ros.actions import Node

def generate_launch_description():
    use_edge_reproj = LaunchConfiguration('use_edge_reproj_factor')

    return LaunchDescription([
        DeclareLaunchArgument('use_edge_reproj_factor', default_value='true'),
        Node(
            package='filter_test',
            executable='graph_optimizer_test',
            name='graph_optimizer_test',
            output='screen',
            parameters=[
                'src/filter_test/config/config.yaml',
                {'use_edge_reproj_factor': use_edge_reproj},
            ],
            arguments=['--ros-args', '--log-level', 'info'],
        ),
    ])

8.5 rclpy 参数事件：Python 工具节点监听配置变化

Jazzy 的 ParameterEventHandler 让 Python 调试工具更容易响应参数变化。 例如调试时动态观察滤波器模型、边缘重投影开关或噪声参数是否被改动。 

import rclpy
from rclpy.node import Node
from rclpy.parameter_event_handler import ParameterEventHandler

class ParamWatch(Node):
    def __init__(self):
        super().__init__('param_watch')
        self.handler = ParameterEventHandler(self)
        self.callback_handle = self.handler.add_parameter_callback(
            parameter_name='use_edge_reproj_factor',
            node_name='/graph_optimizer_test',
            callback=lambda p: self.get_logger().info(f'use_edge_reproj_factor={p.value}'))

rclpy.init()
rclpy.spin(ParamWatch())
rclpy.shutdown()

8.7 切换 RMW 做对比实验

RMW 切换是 ROS 2 很有价值的诊断手段。相同应用代码在不同 RMW 下的发现速度、CPU 占用、 QoS 支持和跨网络表现可能不同。切换时要确保对应 RMW 包已安装。 

# Fast DDS，Humble/Jazzy/Kilted 默认主路径
export RMW_IMPLEMENTATION=rmw_fastrtps_cpp
ros2 run filter_test graph_optimizer_test

# Cyclone DDS，对延迟和发现行为做对比
export RMW_IMPLEMENTATION=rmw_cyclonedds_cpp
ros2 run filter_test graph_optimizer_test

# Kilted 后 rmw_zenoh_cpp 成为 Tier 1，可用于跨网络/边缘网络实验
export RMW_IMPLEMENTATION=rmw_zenoh_cpp
ros2 run filter_test graph_optimizer_test

9. 从 Humble 迁移到 Jazzy/Kilted/Lyrical 的建议

  1. 先升级平台假设：Humble 主平台是 Ubuntu 22.04，Jazzy/Kilted 转向 Ubuntu 24.04。不要只改 ROS_DISTRO，还要检查 OpenCV、GTSAM、Gazebo、编译器和 Python 版本。
  2. 检查头文件和 CMake：保持 include/<package>/... 布局，避免依赖偶然 include path。Kilted 中 ament_target_dependencies 已进入 deprecated 路线，长期应更偏向现代 CMake target。
  3. 把 rosbag2 当成测试工具：迁移前后录同一段仿真数据，比较输出 topic、频率、时间戳和丢包情况。
  4. 明确 RMW 矩阵：至少测试默认 Fast DDS；如果项目有跨机通信或网络复杂环境，再加入 CycloneDDS/Zenoh 对比。
  5. Gazebo 依赖按发行版重审：Jazzy 推荐 Harmonic，Kilted 推荐 Ionic。仿真包不要混用未经验证的 Gazebo/ROS 组合。
  6. 先迁工具节点，再迁实时链路：CLI、launch、rosbag2、RViz、Python 调试工具一般更容易迁；核心滤波和图优化节点最后迁，并用 bag 回放做回归。
对本仓库的直接建议：当前工程仍以 Humble 为准时，不要为了追新版本改动核心跟踪逻辑。 可以先新增 Jazzy/Lyrical 的文档和迁移分支，等 GTSAM、OpenCV、Gazebo 和自定义消息链路都验证后再切主开发环境。
